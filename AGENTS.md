# AGENTS.md

Guidance for future assistant work in this demo.

## Current Result

- JVM test works: `mvn test`.
- JVM smoke works: `mvn -q compile exec:java`.
- Conference plugin JVM smoke works with `mvn -q compile exec:java -Dexec.args="plugins/conference.properties"`.
- Baseline native image builds with `mvn clean -Pnative-baseline -DskipTests package` and is expected to fail at run time with `ClassNotFoundException` for `demo.crema.generated.RuntimeGreeting...`.
- Crema native image builds with `mvn clean -Pcrema -DskipTests package` and the executable successfully loads and invokes Byte Buddy-generated bytecode with `./target/micronaut-bytebuddy-crema Micronaut`.
- The stream demo path is `./demo`: edit `name`, `track`, or `speaker` in `plugins/conference.properties`, press Enter, and show a new `demo.crema.generated.ConferencePlugin...` class loaded after process startup. The native executable also accepts `--watch` without a path and defaults to that plugin file.
- The conference plugin message text is fixed in code. `plugins/conference.properties` intentionally does not have a `message` property.
- Direct native conference smoke works without SLF4J provider warnings: `./target/micronaut-bytebuddy-crema conference`.

## What Worked

- GraalVM CE 25.1.3 exposes `-H:+RuntimeClassLoading`; check with `native-image --expert-options-all | grep RuntimeClassLoading`.
- Micronaut can stay minimal here. `micronaut-runtime` is enough because this is a CLI demo using the application context, not an HTTP server.
- Byte Buddy works in the Crema image when the generated class extends `Object`, defines a public `origin()` method, is loaded with `ClassLoadingStrategy.Default.WRAPPER`, and is invoked with `MethodHandles.publicLookup()`.
- The conference plugin feature works by changing only the `FixedValue` returned from the generated `origin()` method and the generated class prefix. It keeps the same direct method-handle invocation shape as the original working Crema path.
- `org.slf4j:slf4j-nop` silences SLF4J's "No providers were found" warning without enabling log output. Keep this no-op provider unless the demo intentionally needs visible logs.
- `--initialize-at-build-time=net.bytebuddy` is needed. Without it, Byte Buddy fails before runtime class loading with `Java version must be positive: 0` from its VM class-file-version probe.
- Use `mvn clean ... package` for native builds. Repackaging repeatedly without `clean` can make shade-plugin warnings much noisier because the already shaded application jar is reused.

## What Did Not Work

- A generated class extending an AOT application superclass failed in Crema while linking the generated constructor to `demo.crema.RuntimeGreeting.<init>()V`.
- Reflective constructor lookup on the runtime-loaded class failed with `NoSuchMethodException`; method handles worked better for this demo.
- Interface/default-method dispatch to the generated implementation was not stable. It produced `AbstractMethodError` or an uncompiled-method-pointer abort, depending on the shape.
- Do not refactor the plugin demo to use a generated interface implementation for dispatch unless retesting the native executable. The direct generated `origin()` method is the known working path.
- The baseline native image is not supposed to run successfully. Its failure is the contrast case showing what `-H:+RuntimeClassLoading` changes.

## Notes

- The Crema flag is experimental on this GraalVM build; native-image warns that `-H:+UnlockExperimentalVMOptions` may be required in the future.
- The Micronaut parent currently emits a Maven model warning about duplicate `flexmark-html2md-converter` dependency-management entries. It did not block the verified builds.
